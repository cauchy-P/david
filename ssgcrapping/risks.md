좋아, 너희 팀이 실제 회의에서 썼을 법한 톤으로, PRD와 MVP 스코프에 기반한 **리스크 / 제약 목록**을 싹 정리해줄게.
일정, 법, 보안, 데이터 의존성, 기술적 문제 등 범주별로 나눴고 “왜 리스크인지”까지 구체적으로 넣었어.

---

# 리스크 및 제약 목록

---

## 1. 일정 관련 리스크

### 1) **3주 개발 기간 내 기능 범위를 모두 구현하지 못할 가능성**

* 이유: 사이트별 DOM 구조가 다르고 예외 처리가 많아 예상보다 시간이 추가될 수 있음.
* 영향: 일부 사이트 크롤링 실패, 중복 병합 미완료, 대시보드 제외 등 스코프 축소 가능.

### 2) **사이트 구조 변경으로 재작업 발생**

* 이유: 크롤링 대상 사이트가 수시로 UI/HTML을 바꾸면 1~2일 재작업 필요.
* 영향: 일정 지연, 안정성 KPI 저하.

### 3) **테스트에 충분한 시간이 부족할 수 있음**

* 이유: 크롤러는 “한 번 성공”이 아니라 “지속적으로 성공해야” MVP 기준 충족.
* 영향: 실제 운영 단계에서 오류 발생 가능성 증가.

---

## 2. 기술적 리스크

### 1) **반(半)자동 구조로 인해 안정성 낮을 가능성**

* 크롬 익스텐션 기반 수집을 배제하긴 했지만, 서버 크롤링 역시 반자동 규칙 기반이라 실패 확률이 높음.

### 2) **중복 병합 로직 정확도 낮음**

* 행사 제목, 날짜 표현식이 사이트별로 상이해 rule-based 매칭이 어렵다.
* KPI인 “중복 병합 정확도 ≥ 90%” 달성 실패 위험.

### 3) **필수 필드 누락 문제**

* 날짜나 가격이 이미지로만 제공되는 사이트 등 parsing 불가능한 경우 존재.
* KPI(필수 필드 누락률 ≤ 5%) 충족 어려움.

### 4) **사이트 구조 복잡도 차이**

* iframe, 동적 JS 렌더링, infinite scroll 등이 있을 수 있음.
* 서버 크롤링만으로는 수집 불가한 페이지 발생.

---

## 3. 데이터 관련 리스크

### 1) **데이터 품질 편차**

* 사이트에 따라 정보가 미흡하거나 복잡해 JSON 스키마에 넣기 어려움.
* 데이터 정제 과정이 늘어남.

### 2) **데이터 업데이트 지연**

* 사이트에 새로운 이벤트가 올라오더라도 크론 주기(6시간 간격) 안에서는 딜레이 발생.
* “수집 지연 시간 ≤ 6시간” KPI 영향.

### 3) **데이터 중복 탐지 오류**

* 동일 행사라도 시간·가격이 조금씩 다르게 표기되면 다른 행사로 오인 가능.

### 4) **수집 데이터 편향**

* 초기 MVP는 소수 웹사이트만 대상이라 시장 전체 행사 트렌드를 반영하지 못함.

---

## 4. 법적 리스크

### 1) **웹 크롤링의 법적 모호성**

* 일부 사이트는 봇 크롤링을 금지하는 이용약관을 갖고 있음.
* 법적으로는 회색지대지만, 민원이나 요청이 들어오면 수집이 중단될 수도 있음.

### 2) **robots.txt 정책 위반 가능성**

* 스케줄러가 자동으로 돌면서 robot 정책을 위반할 가능성 존재.
* 비즈니스 운영 시 법무 검토 필요.

### 3) **데이터 저작권 문제**

* 행사 정보(텍스트, 이미지)의 저작권 문제 발생 가능.
* 단순 텍스트 정보는 보통 자유롭게 사용되지만 이미지 사용은 제한적.

---

## 5. 보안 리스크

### 1) **외부 사이트 연결에 따른 보안 취약성**

* 크롤링 요청 중 악성 script, 비정상 응답을 처리해야 할 필요가 있음.

### 2) **서버에 저장되는 데이터의 보안 문제**

* 행사 정보는 민감 데이터는 아니지만, 로그/URL에 사용자 계정 정보가 포함될 경우 위험.

### 3) **확장 프로그램 사용 시 보안 이슈**

* MVP에서 제외됐지만 향후 확장 프로그램을 도입할 경우 개인정보 접근 권한 문제가 발생할 수 있음.

---

## 6. 운영 및 유지보수 리스크

### 1) **사이트마다 유지보수 비용이 상이**

* 3~5개 사이트만 대응하더라도 유지보수는 누적됨.
* 사이트가 늘어날수록 비용이 기하급수적으로 증가.

### 2) **에러 모니터링 체계 부족**

* MVP에서는 error alert 시스템을 간소화할 가능성이 높음.
* 운영 중 문제 발생을 즉시 파악하기 어려움.

### 3) **서버 리소스 비용 증가**

* 크롤링은 트래픽을 많이 사용하며, 주기적 실행으로 서버 비용이 증가할 가능성.

---

## 7. 제품 관점의 제약

### 1) **추천 기능이 범위에서 제외**

* 핵심 사용자 가치("나에게 맞는 행사 추천")를 완전 해결하지 못함.
* 초기 사용자 만족도 낮아질 수 있음.

### 2) **대시보드 기능이 제한적**

* MVP는 내부용 데이터 수집이 목적이므로 외부 UX는 매우 단순.
* 기획자 입장에서는 "거의 그냥 데이터 덤프 수준"일 수 있음.

### 3) **분류/태깅 기능 미구현**

* 행사 카테고리(전시/공연/클래스 등) 부재
* 실제 분석 단계에서 활용도가 떨어질 수 있음.

---

## 8. 팀 역량 및 프로세스 리스크

### 1) **크롤링 경험 부족**

* 팀 구성원이 크롤링 경험이 적으면 예외 케이스 처리에 시간이 많이 들어감.

### 2) **테스트 자동화 미흡**

* 정기 수집 시스템은 자동 테스트가 없으면 품질 검증이 어려움.

### 3) **지속 운영 인력 부족**

* MVP 이후 운영이 가능한 인력이 없으면 금방 망가질 수 있음.
