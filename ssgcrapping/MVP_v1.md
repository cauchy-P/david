좋아, 지금 PRD 흐름에 맞춰서 **일정 기반 MVP 스코프 표 + 우선순위 근거 노트**를 바로 만들었어.
3주짜리 프로젝트라는 조건을 전제로 현실적인 스코프만 잡았고, 왜 이 기능이 먼저인지도 짧고 명확하게 이유까지 덧붙였어.

# ✅ 일정 기반 MVP 스코프 표 (3주)

아래 표는 "개발 일정 기준"으로 정리했어.
주차별 목표와 구체 기능을 단계적으로 나눈 형태.

---

## **1주차 — 기반 구축 및 1차 수집 성공**

| 기능                  | 설명                          | 우선순위  | 비고             |
| ------------------- | --------------------------- | ----- | -------------- |
| 수집 대상 사이트 선정 (3~5개) | 스키마 구조 정의 및 초기 대상 확정        | ★★★★★ | 전체 크롤링 구조의 기준점 |
| 데이터 스키마 정의(JSON/DB) | 필드 정의(제목, 날짜, 장소, 링크, 가격 등) | ★★★★★ | 품질 기준의 핵심      |
| 기본 크롤링 엔진 구축        | HTML 파싱, 요청/응답 구조           | ★★★★★ | MVP의 근간 기능     |
| 1개 사이트 크롤링 성공       | “최소 기능성 확보 목표”              | ★★★★★ | 1주차 핵심 완료 조건   |
| 로그/에러 기록 기능         | 크롤링 실패 원인 추적                | ★★★★☆ | 안정성 확보         |

---

## **2주차 — 확장 및 정제 기능 개발**

| 기능                          | 설명             | 우선순위  | 비고             |
| --------------------------- | -------------- | ----- | -------------- |
| 추가 사이트 크롤링 (3~5개)           | 구조 다른 사이트 대응   | ★★★★★ | 스케일링의 핵심       |
| 정기 실행 스케줄러                  | 크론 기반 자동 수집    | ★★★★★ | “자동 수집” KPI 충족 |
| 중복 식별 및 병합 로직               | 동일 행사 여러 링크 병합 | ★★★★☆ | 데이터 품질 핵심      |
| 필수 필드 검증 로직                 | 누락률 관리         | ★★★★☆ | KPI 품질 기준과 연계  |
| 데이터 저장소 구축(DB or JSON 스토리지) | 누적 데이터 관리      | ★★★★☆ | 후처리 기반         |

## **3주차 — 정리, 적재, 전달 기능**

| 기능                       | 설명                | 우선순위  | 비고            |
| ------------------------ | ----------------- | ----- | ------------- |
| 엑셀 변환 기능                 | 담당자 전달을 위한 필수 기능  | ★★★★★ | Must Have에 해당 |
| 단순 대시보드 (선택적)            | 수집량, 오류 등 기본 표기   | ★★☆☆☆ | Should Have   |
| 중복 처리 검증 및 품질 개선         | 병합 정확도 튜닝         | ★★★★☆ | KPI 충족        |
| 성능 안정화 및 예외 처리           | 사이트별 비정형 HTML 대응  | ★★★★☆ | 마지막 QA 단계     |
| 프로젝트 리포트 자동 생성(optional) | 추후 internal 사용 목적 | ★☆☆☆☆ | MVP는 아님       |

# ✅ 우선순위 근거 노트

아래는 MVP 관점에서 "왜 이 기능이 반드시 먼저여야 하는가"를 설명하는 메모.

---

## **1) 자동 수집(크롤링) 기능이 절대 핵심**

* PRD의 성공 지표 70% 이상이 “자동 수집”과 직결됨
* 추천/대시보드 등 후속 기능은 완전히 부차적
* 사이트 한 곳이라도 정확히 크롤링해야 다음 단계가 가능
  → 그래서 1주차 목표는 “1개 사이트 완전 동작”으로 설정함

---

## **2) 데이터 스키마 정의가 품질의 기준**

* 스키마가 잘못되면 이후 기능(병합, 품질 체크, 엑셀 출력)이 모두 부정확
* 필수 필드 누락률 KPI도 스키마 없이는 체크 불가
  → 초기 스키마 정의는 무조건 1주차 최우선

---

## **3) 여러 사이트 대응이 진짜 난이도**

* 사이트마다 구조가 다르고, 이게 크롤링 실패의 핵심 원인
* PRD에서 “돈 없는 우리는 ML이 아니라 규칙 기반 접근” 명시
  → 2주차 목표는 사이트 확장에 대부분의 시간을 배정

---

## **4) 중복 병합은 데이터 품질의 핵심**

* “행사 하나에 링크가 여러 개 있는 문제”는 실제 담당자들이 가장 귀찮아하는 지점
* Must Have 항목은 아니지만 품질 지표(KPI)와 직결됨
  → 2주차 후반 배치한 이유는 크롤링 로직이 먼저 있어야 하기 때문

---

## **5) 엑셀 변환 기능은 실제 사용자에게 필요한 최종 산출물**

* 담당자가 실제 업무에서 쓰는 형태는 엑셀
* MVP의 최종 전달물은 데이터베이스가 아니라 “엑셀 파일”
  → 3주차의 가장 중요한 기능

---

## **6) 대시보드는 있어도 되고 없어도 됨**

* Should Have로 설정됨
* 핵심 목표는 자동 수집이지, UI는 아님
  → 3주차 후반, 시간 남으면 구현

---

## **7) 안정화/예외처리가 MVP 품질을 좌우**

* 스케줄러가 10번 중 3번 실패하면 MVP가 무너짐
* “크롤링 실행 성공률 95%” KPI를 만족시키려면 마지막 주에 튜닝 필수
  → 3주차 후반 QA를 명확히 일정에 포함

---

필요하면 이 범위를 기반으로 **3주 개발 캘린더(Google Calendar 배정 버전)**나 **Jira 티켓 단위 상세 스프린트 플랜**도 만들어줄게.
