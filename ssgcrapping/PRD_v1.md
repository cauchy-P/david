
# PRD v1 (제품 요구사항 정의서)

## 1. 문제 정의

사람들이 행사 정보를 얻기 위해서는 뉴스레터 구독, SNS 탐색, 행사 사이트 방문 등 여러 채널을 직접 뒤져야 한다. 이는 반복적이고 귀찮으며, 일정에 맞는 행사를 찾기 위해 사용자가 많은 ‘발품’을 팔아야 한다. 특히 공연/행사 기획 담당자들은 여러 사이트를 주기적으로 확인해야 해 업무 스트레스가 크다.
이를 해결하기 위해 **여러 행사 사이트에서 정보를 자동으로 수집하여 데이터베이스화하는 시스템**이 필요하다.

---

## 2. 페르소나

### 1) 일반 사용자

* 특징: 새로운 경험, 재미있는 이벤트, 전시·공연을 찾고 싶지만 정보를 찾아다니는 것이 귀찮음
* 현재 행동: 인스타그램, 포스터, 뉴스레터, 행사 플랫폼 등을 직접 탐색
* 페인포인트:

  * 원하는 일정에 맞는 행사를 찾기 어려움
  * 지속적인 스크롤링이 귀찮음
  * 함께 갈 친구의 부재, 일정 미스 등으로 동기 떨어짐
  * 행사 종류가 너무 많아 선택 피로도 존재

### 2) 공연/행사 기획 담당자

* 특징: 일정을 모니터링하고 참고하기 위해 경쟁사·시장 행사 정보를 지속적으로 확인해야 함
* 페인포인트:

  * 사이트별 업데이트 구조가 달라 반복적으로 방문해야 함
  * 정확한 일정/장소/티켓 정보 파악이 매번 번거로움
  * 모니터링을 자동화해주는 도구가 없음

---

## 3. 사용자 시나리오

1. 담당자는 프로젝트의 크롤링 서버에 수집 대상 사이트(예: 3~5개)를 미리 등록한다.
2. 크롤러는 정해진 주기로 사이트를 방문해 행사 정보를 자동 수집한다.
3. 수집된 데이터는 JSON/데이터베이스 형식으로 정리되어 저장된다.
4. 저장된 데이터는 엑셀로 변환하거나 리뷰 단계에서 확인할 수 있다.
5. 중복된 행사 정보(사이트별 중복)는 자동 병합하여 하나의 이벤트로 통합된다.
6. 담당자는 축적된 데이터를 기반으로 일정/시장 트렌드를 확인한다.

---

## 4. 제품 요구사항

### 4.1 기능 요구사항 (Must Have)

* 정기적으로 크롤링을 수행하는 서버 구축
* 타겟 사이트에서 행사 정보(제목, 날짜, 장소, 가격, 링크 등) 자동 수집
* 수집된 데이터의 필수 필드 검증
* 중복 행사 자동 감지 및 병합
* JSON/DB 형태로 저장
* 데이터 엑셀 파일 생성 기능
* 크롤링 실패 시 로그 기록 기능

### Should Have

* 간단한 대시보드 (수집량, 오류, 최신 데이터 확인 정도)
* 크롬 확장 프로그램 기반의 보조 수집 기능(초기 MVP에선 생략 가능)
* 사이트별 수집 우선순위 설정

### Won’t Have

* 외부 사용자 대상 추천 시스템
* UI/검색 기능이 있는 완전한 데이터베이스 서비스
* ML 기반 자동 태깅 기능

---

## 5. 비기능 요구사항

* **안정성**: 크롤러는 최소 24시간 기준 95% 이상 정상 실행
* **확장성**: 신규 사이트 추가 시 3시간 이내 스키마/메타데이터 등록 가능
* **데이터 신뢰성**: 필수 필드 누락률 5% 이하
* **중복 처리 정확성**: 동일 행사 인식률 90% 이상
* **수집 주기**: 기본 6시간 간격으로 자동 실행
* **로그 기록**: 모든 에러 및 수집 기록은 서버에 저장

---

## 6. KPI / 성공 기준

### 주요 KPI

* 자동 크롤링 성공률(≥ 95%)
* 필수 정보 정확도(≥ 90%)
* 필수 필드 누락률(≤ 5%)
* 일정 기간 내 수집량(예: 3주간 300건 이상)
* 중복 병합 정확도(≥ 90%)
* 신규 사이트 추가 소요시간(≤ 3시간)
* 수집 지연 시간(사이트 업데이트 후 데이터 반영까지 평균 ≤ 6시간)

### 보조 KPI

* 담당자의 모니터링 시간 감소율
* 엑셀 변환 성공률 100%

---

## 7. 범위 및 스코프 정의

* **범위 포함**

  * 특정 사이트 목록(3~5개)에서 행사 정보를 자동 수집
  * 서버 기반 주기 크롤링
  * 데이터 정제 및 저장
  * 중복 병합 처리
  * 엑셀 파일 출력

* **범위 제외**

  * 추천 알고리즘
  * 실시간 알림 서비스
  * 태그 자동 생성
  * 대시보드 고도화
  * 크롬 확장 프로그램 1차 버전에서는 제외

---

## 8. 리스크

* 사이트 구조 변화로 인한 크롤링 실패
* 특정 사이트의 anti-scraping 정책에 따른 차단
* JSON 스키마의 변경 필요
* 중복 판단 기준의 불확실성
* 수집량이 적을 경우 MVP의 효과가 낮아질 수 있음

---

필요하면 이 PRD를 바탕으로 **MVP 기능 체크리스트**, **데이터 스키마 설계**, **3주 개발 타임라인**까지 깔끔하게 정리해서 만들어줄게.
